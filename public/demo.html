A queryable context layer that makes design systems machine-readable, enforceable, and usable by AI — via the open Model Context Protocol (MCP).

What is a Design System MCP?

Instead of pasting documentation into a prompt, you expose your design system as a set of queryable tools. An AI assistant can then look up exact token values, fetch component constraints, validate its own output against the real system, and detect CSS values that don't match any token — all at query time, not from a snapshot.

Because MCP is an open standard, one server works with Claude Desktop, Cursor, GitHub Copilot, and any custom agent — without rewriting function schemas for each platform.

Why not just use JSON or a REST API?

ApproachKey limitations
Paste docs into a promptSnapshot in time; eats context window; AI guesses token names
Static JSON in contextEntire file in every prompt; no selectivity; still a snapshot
Database / REST APICustom function schema per AI platform; no standard protocol
Figma Variables / Tokens StudioNot directly callable by AI tools without a custom translation layer
MCP ✓One server, any MCP-compatible client — live, selective, validated

Why this matters

Without MCPWith MCP
AI reads static docs pasted into a promptAI fetches structured data at query time
Token names are guessed or hallucinatedToken names come from the live source of truth
Component constraints exist only as proseConstraints are machine-readable and enforceable
Validation happens in code reviewValidation happens before code is written
Design system changes require re-promptingAI always queries the current state of the system

How it works

User  ──▶  LLM (AI)  ──▶  MCP Server (POST /mcp)  ──▶  Design System
                  ◀── grounded response  ◀── structured data (tokens, components, themes)
  1. User describes a UI — e.g. "Create a login form with a primary submit button"
  2. The LLM recognises it needs design system context and calls MCP tools automatically (get_component("button"), get_tokens("spacing"))
  3. The MCP server returns real token values, approved variants, constraints, and accessibility rules
  4. The LLM generates a grounded response using actual token names — not guesses
  5. The Live Preview renders the result; the MCP Tools panel shows which queries were made
